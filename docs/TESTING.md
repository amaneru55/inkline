# InkNest 测试策略

测试目标是让核心阅读链路、数据模型、任务系统和 UI 交互可以长期演进。覆盖率目标尽量接近 100%，尤其是纯逻辑层和关键交互层。

## 覆盖率目标

- `core` 层：接近 100%。
- React hooks：接近 100%。
- 关键 UI 组件：覆盖主要状态和交互。
- 页面层：覆盖关键流程，不追求重复测试所有组合。
- Rust 核心模块：覆盖文件解析、路径处理、数据库迁移、任务状态和错误处理。

覆盖率下降必须有明确原因。性能关键路径、隐私边界和数据迁移不能无测试合并。

## 前端测试

使用 Vitest。

重点测试：

- 领域模型转换。
- 自然排序和页序计算。
- 阅读模式状态机。
- 双页、长条、右到左和左到右布局计算。
- TanStack query key 和 command client 类型边界。
- hooks 的加载、错误、取消和重试状态。
- i18n fallback 和格式化。

## Storybook 测试

所有 UI 组件都需要 story。复杂组件需要 interaction tests。

重点覆盖：

- Reader toolbar。
- Page viewport。
- Library item card/list row。
- Task queue item。
- Settings controls。
- Theme switcher。
- Private Vault 相关入口和状态。

Storybook 既是视觉文档，也是回归测试入口。

## 性能测试

性能测试从第一阶段就建立基准。

需要记录：

- 应用启动时间。
- 本地目录扫描时间。
- 压缩包页列表读取时间。
- 当前页到下一页的可见延迟。
- 图片预加载命中率。
- 阅读器滚动帧率。
- 内存占用和图片释放行为。

阅读器交互变更需要手动或自动验证不引入明显卡顿。

## Rust 测试

Rust 侧重点：

- 路径规范化。
- 压缩包条目过滤和自然排序。
- 数据库 schema 和迁移。
- 任务状态转换。
- 缓存清理策略。
- 脱敏日志。

涉及真实文件的测试使用 fixtures，避免依赖用户本地路径。

## 测试数据

需要逐步建立 fixtures：

- 图片文件夹。
- zip/cbz 示例。
- 嵌套目录。
- 异常排序文件名。
- 缺页、重复页、损坏文件。
- 多语言 UI 文案。
